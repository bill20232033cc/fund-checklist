## Findings

### 001-未修复-严重-产品定位仍锁定为自动结构化报告系统，不是 Agent 文档阅读工具

- **入口/函数**: `fund-analysis analyze` / `FundAnalysisService.analyze()` / `_run_analysis_core()`
- **文件(行号)**: `README.md:1-5`, `README.md:153-177`, `fund_agent/ui/cli.py:651-758`, `fund_agent/services/fund_analysis_service.py:665-717`, `fund_agent/services/fund_analysis_service.py:1089-1116`
- **输入场景**: 用户需要 MVP 级 `PDF -> Docling JSON -> ToolService -> Agent tools`，让 Agent 阅读、搜索、定位 PDF 内容。
- **实际分支**: CLI 默认进入 `analyze/checklist/analyze-annual-period`，Service 再进入结构化抽取、quality gate、模板渲染、程序审计。
- **预期行为**: MVP 主线应暴露文档工具能力：列报告、列章节、读章节、搜索、列表格、读表格、取 excerpt，并把自动报告/quality gate/golden/readiness 放到后置路径。
- **实际行为**: README 首屏把项目定义为生成 8 章体检报告；当前能力清单继续列 `analyze`、`checklist`、字段评分、golden、quality gate、preflight；真实 Service 入口仍先调用 `_extractor.extract()`，没有文档阅读 ToolService 主路径。
- **直接证据**: `README.md:3` 声称生成 8 章 Markdown 体检报告；`README.md:157-177` 把主能力列为 analyze/checklist/golden/quality gate；`fund_agent/services/fund_analysis_service.py:681-699` 渲染模板并执行程序审计；`fund_agent/services/fund_analysis_service.py:1111-1115` 核心路径先抽取 `StructuredFundDataBundle`。
- **影响**: 开发目标被自动判断/报告生产牵引，任何 live/PDF/network 验证都会被 quality gate、golden、source-truth、final judgment 等非 MVP 阅读能力阻断，导致持续打补丁而不是形成可上线的 Agent 工具。
- **建议改法和验证点**: 新主线先定义并实现 `FundDocumentToolService`，只承诺文档读取工具契约；默认 CLI/Service 验证改为 tool smoke，而不是 8 章报告通过。验证点：同一 PDF 能完成 `list_sections/read_section/search_document/list_tables/read_table/get_excerpt`，返回稳定 locator 和 safe excerpt。
- **修复风险（低/中/高）**: 高。需要重置产品面和 README/设计真源，而不是局部修函数。
- **严重程度（低/中/高/严重）**: 严重

### 002-未修复-严重-Docling 被设计成 candidate evidence，而不是生产文档中间层

- **入口/函数**: `FundDocumentRepository.load_annual_report()` / `AnnualReportPdfAdapter` / `FundDataExtractor.extract()`
- **文件(行号)**: `AGENTS.md:75-83`, `docs/implementation-control.md:29-44`, `fund_agent/fund/documents/repository.py:1-5`, `fund_agent/fund/documents/adapters/annual_report_pdf.py:169-200`, `fund_agent/fund/pdf/parser.py:185-220`, `fund_agent/fund/documents/candidates/representation_handlers.py:1-7`
- **输入场景**: 生产读取基金年报 PDF，希望一次转换为 Docling JSON 并由 ToolService 消费。
- **实际分支**: `FundDocumentRepository` 默认返回 `ParsedAnnualReport`；PDF adapter 使用 `extract_text/extract_tables/locate_sections`，底层 parser 直接调用 `pdfplumber`；Docling 代码仅存在于 `documents/candidates`，明确不接入生产仓库和 Service/UI/Host/renderer/quality gate。
- **预期行为**: Docling JSON 应成为读取工具的主要可复用中间层，生产仓库或文档 store 应能持久化和读取 Docling document，并由 ToolService 做受控查询。
- **实际行为**: Docling 被降级为候选 evidence harness；生产路径仍是 EID PDF -> pdfplumber -> ParsedAnnualReport -> extractor。
- **直接证据**: `repository.py:3-5` 写明仓库复用 PDF 适配器下载与解析；`annual_report_pdf.py:177-180` 默认注入 `extract_text/extract_tables/locate_sections`；`pdf/parser.py:11` 直接依赖 `pdfplumber`；`representation_handlers.py:3-6` 明确 Docling/pdfplumber/EID HTML 候选表示“不接入生产 FundDocumentRepository”。`docs/implementation-control.md:44` 进一步禁止 Service/UI/Host/renderer/quality gate 直接消费 Docling。
- **影响**: 与目标 `PDF -> Docling JSON -> ToolService` 方向相反。继续沿此路径开发，只会加深 `ParsedAnnualReport`、Extractor、accepted facts、quality gate 的路径依赖。
- **建议改法和验证点**: 新建独立 production document path：`PdfSourceProvider -> PdfBlobStore -> DoclingConverter -> DoclingDocumentStore -> FundDocumentToolService`。保留旧 `ParsedAnnualReport` 作为 legacy report path，禁止它决定新阅读工具架构。验证点：不导入 `pdfplumber` 的读取工具测试通过；Docling JSON fixture 可独立驱动章节/表格/搜索工具。
- **修复风险（低/中/高）**: 高。需要迁移文档仓库契约与测试分层。
- **严重程度（低/中/高/严重）**: 严重

### 003-未修复-高-仓库不存在当前 MVP 所需的 Document ToolService/API surface

- **入口/函数**: public package/API surface
- **文件(行号)**: `docs/design.md:532-539`, `fund_agent/fund/README.md:131`, `fund_agent/fund/documents/candidates/representation_handlers.py:64-87`
- **输入场景**: Agent 需要通过工具读取 PDF：列 sections、读 section、搜索文本、读表格、取引用片段。
- **实际分支**: `rg` 只找到未来候选名 `FundToolService`，并且 design 明确它不是当前代码类型；候选 handler 只提供 representation route handlers，不是 Agent tool API。
- **预期行为**: 应存在稳定 ToolService 类型和测试，例如 `FundDocumentToolService.list_sections()`、`read_section()`、`search_document()`、`list_tables()`、`read_table()`、`get_excerpt()`。
- **实际行为**: 没有可调用的文档工具服务；当前对外入口仍是结构化 bundle、报告、quality gate、golden 和 LLM report path。
- **直接证据**: `docs/design.md:539` 写明 `FundToolService` 仍是未来候选命名，不是当前代码类型；`fund_agent/fund/README.md:131` 也说相关能力不是 `FundToolService`；`representation_handlers.py:64-87` 只是构造 candidate route handler map。
- **影响**: Agent 无法以 dayu fins 类似方式“读文档、查信息”；只能被迫走结构化字段抽取或报告生成。MVP 验收对象缺失，导致测试只能验证旧系统没有坏，不能证明新产品可用。
- **建议改法和验证点**: 先实现最小 `FundDocumentToolService`，再把 CLI/API smoke 建在该 surface 上。验证点：一个 Docling JSON fixture + 一个 live PDF 转换样本均可返回稳定 locator；缺章节/缺表格返回可预期错误码，不进入 quality gate。
- **修复风险（低/中/高）**: 中。可在新模块旁路实现，避免先大规模删除旧报告系统。
- **严重程度（低/中/高/严重）**: 高

- **实际行为**: 当前测试多数只证明 fake repository / in-memory bundle / retained excerpt / no-live candidate helpers；没有围绕 Document ToolService 的 live/PDF/network acceptance matrix。

- **影响**: 测试通过不能提升为 MVP ready，只能说明旧报告链路的局部契约稳定。继续堆 no-live 单元测试会扩大维护面，却不降低上线风险。

- **修复风险（低/中/高）**: 中。需要定义新验收矩阵，但可以与旧测试并存。
- **严重程度（低/中/高/严重）**: 高
