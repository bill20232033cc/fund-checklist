### 001-未修复-严重-开发流程（Gateflow）已反噬产品，428 commits 中 75% 是流程管理

- **入口/函数**: 整个 git history
- **文件(行号)**: git log
- **输入场景**: 任何开发任务进入 gateflow 流程
- **实际分支**: 428 commits 中 323 个是 gateflow/gate/review/control/sync/accept/closeout/disposition/evidence/checkpoint 类型，仅约 105 个是实际功能实现
- **预期行为**: 开发流程应服务于产品交付，流程开销应远小于功能实现
- **实际行为**: 每个功能点被拆成 plan → review (DS+MiMo) → evidence → controller judgment → re-evidence → re-review → closeout → control sync 的 8+ 步流程，每步 2-5 个 review 文件。一个 S6 字段族 selector 需要 S6-A 到 S6-G 七个子 gate，每个子 gate 又有 plan/review/evidence/controller judgment
- **直接证据**: `git log --oneline | grep -i "gateflow\|gate\|review\|control\|sync\|accept\|closeout\|disposition\|evidence\|checkpoint" | wc -l` = 323；`git log --oneline | grep -iv "gateflow\|gate\|review\|control\|sync\|accept\|closeout\|disposition\|evidence\|checkpoint" | wc -l` ≈ 105
- **影响**: 开发速度被流程本身吞噬。Agent 每次推进一个 gate 都需要读取 implementation-control.md（566 行）、current-startup-packet.md（228 行）和大量历史 review 文件，大部分 token 花在理解"当前在哪"而不是"做什么"
- **建议改法和验证点**: 砍掉 gateflow 流程，改为直接实现 → 测试 → merge 的轻量循环。控制文档压缩为一页：产品目标、当前阶段、下一步、non-goals
- **修复风险（低/中/高）**: 高。需要重置开发文化
- **严重程度（低/中/高/严重）**: 严重

### 002-未修复-严重-产品定位错误：自动报告系统 vs Agent 文档阅读工具

- **入口/函数**: README.md / design.md / FundAnalysisService
- **文件(行号)**: README.md:1-5, design.md:17-19
- **输入场景**: 用户或 Agent 需要阅读基金年报
- **实际分支**: 项目被定义为"生成 8 章 Markdown 体检报告"的自动分析系统

- **直接证据**: README.md:3 声称"生成 8 章 Markdown 体检报告"；design.md:17 北极星是"让普通基金投资者获得专业级基金体检报告"；当前 CLI 入口全部是 analyze/checklist/golden/quality-gate
- **影响**: 所有后续开发（Docling、source-truth、candidate evidence、quality gate、golden readiness）都建立在"自动判断"这个前提上，但这个前提本身需要大量未证明的假设（parser 正确性、字段完整性、LLM 输出质量）
- **建议改法和验证点**: 重新定义北极星为"让 Agent 能读懂基金年报"。MVP = PDF → Docling JSON → ToolService（list_sections / read_section / search / list_tables / read_table / get_excerpt）。自动报告降级为 ToolService 之上的高阶应用
- **修复风险（低/中/高）**: 高。需要重写 README/design/AGENTS.md 的产品定义
- **严重程度（低/中/高/严重）**: 严重

## Root Cause 分析

以上 8 个 findings 的共同 root cause 是：

**项目在没有清晰 MVP 定义的情况下启动了过度工程化的开发流程。**

具体表现为：

1. **MVP 定义缺失**：项目从未明确回答"第一个可用版本能做什么"。北极星是"让普通投资者获得专业级体检报告"，但这不是一个 MVP——这是一个终极目标。MVP 应该是"让 Agent 能读懂一份年报"。

2. **流程先于产品**：Gateflow 流程在产品定义清晰之前就被引入。结果是每个开发决策都需要走 plan → review → evidence → judgment 的完整流程，但这些流程验证的是"旧路线是否完备"，而不是"新产品是否可用"。

3. **验证无限递归**：因为产品是"自动判断报告"，每个判断环节（parser、字段、LLM、审计）都需要验证，每个验证又发现新的未覆盖边界，形成死循环。如果产品是"文档阅读工具"，验证标准就是"能不能读"，这是有限的。

4. **控制文档成为产品**：566 行 implementation-control + 228 行 startup-packet + 数百个 review 文件，这些文档本身成为了开发的主要产出物，而不是产品代码。
